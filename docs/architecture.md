# T.O.T. 系統架構文件

## 概觀

T.O.T. (The Octo-Teller) 是一個 Telegram 上的多人 AI 地下城主系統。
核心設計理念是「哭哭鱆雙形態」：骷髏（確定性規則）與章魚（LLM 敘事）分離，
確保規則判定的可靠性，同時保有敘事的創造力。

## 核心設計原則

### 1. 骷髏與章魚分離

| 元件 | 類型 | 說明 |
|------|------|------|
| Bone Engine | 確定性 | 骰子、戰鬥、法術、狀態 — 不依賴 LLM，結果可預測可測試 |
| Narrator | LLM 驅動 | 場景描述、NPC 對話、劇情 — 依賴 Anthropic Claude API |
| Mimic | LLM 驅動 | 自然語言解析 — 將玩家輸入轉為結構化 GameAction |
| Companion | 混合 | AI 隊友 — 確定性戰術評估 + LLM 個性化對話與判斷 |
| Extension | LLM 驅動 | 脫稿處理 — 處理規則外的即興情境 |

**重要**：所有 LLM 呼叫只在 Narrator / Mimic / Companion / Extension 中。Bone Engine 絕不呼叫 LLM。

### 2. 玩家動作處理流程

```
玩家輸入 "我要用火球術攻擊哥布林群"
         │
         ▼
    ┌─────────┐
    │  Mimic   │  解析意圖 → CastSpellAction(spell="Fireball", target="goblin_group")
    └────┬────┘
         ▼
    ┌────────────┐
    │ Bone Engine │  規則判定 → 法術位檢查、DC 計算、骰傷害、套用效果
    └────┬───────┘
         ▼
    ┌──────────┐
    │ Narrator  │  生成敘事 → "火焰從你掌心爆發，哥布林群在烈焰中慘叫..."
    └──────────┘
         │
         ▼
    Telegram 回覆玩家
```

### 3. 資料流向

所有跨模組資料傳遞使用 **Pydantic model**：

- `GameAction` — Mimic 輸出，Bone Engine 輸入
- `RuleResult` — Bone Engine 輸出，Narrator 輸入
- `NarrativeResponse` — Narrator 輸出，Bot 層格式化後傳送

## 六隻 Gremlin 詳解

### Prep（備戰精靈）

負責遊戲開始前的準備工作：
- 世界生成（地圖、NPC、據點）
- Session 準備（回顧上次進度、設定場景）
- 遭遇生成（隨機遭遇表、難度平衡）

### Bone Engine（骷髏引擎）

確定性規則引擎，處理所有 D&D 2024 (5.5e) 機械判定：
- **dice.py** — 骰子系統（d4~d100、advantage/disadvantage、modifier）
- **combat.py** — 戰鬥引擎（先攻、回合制、攻擊/傷害、機會攻擊）
- **spells.py** — 法術系統（施法、法術位消耗、專注維持）
- **conditions.py** — 狀態系統（12 種基礎狀態、堆疊規則、持續時間）
- **character.py** — 角色系統（屬性、技能、生命值、等級）
- **rest.py** — 休息機制（短休/長休、HP/法術位恢復）

### Narrator（章魚說書人）

LLM 驅動的敘事引擎：
- 根據 Bone Engine 的判定結果生成場景描述
- 支援 6 種敘事風格（史詩、幽默、恐怖、寫實、童話、極簡）
- 難度自適應（根據玩家經驗調整描述深度）
- NPC 語音特色（每個 NPC 有獨特說話方式）

### Mimic（擬態解析器）

自然語言 → 結構化動作轉換：
- 意圖分類（攻擊、施法、對話、移動、查詢、系統指令）
- 模式匹配 + LLM 混合解析
- 輸出標準化 GameAction Pydantic model

### Companion（AI 隊友）

自主決策的 AI 同伴系統，讓隊友像真人一樣有自己的想法：

**核心機制：信任驅動的自主決策**

```
玩家下達戰術建議 "幫我治療"
         │
         ▼
  ┌──────────────┐
  │  信任度評估    │  玩家過去的建議帶來好結果？→ 信任度高
  └──────┬───────┘
         ▼
  ┌──────────────┐
  │  情境評估      │  自己 HP 也很低？隊友更危險？敵人快死了？
  └──────┬───────┘
         ▼
  ┌──────────────┐
  │  個性權衡      │  保護型 → 傾向治療 / 好戰型 → 傾向先殺敵
  └──────┬───────┘
         ▼
  ┌──────────────┐
  │  最終決策      │  聽從建議 / 部分採納 / 拒絕並說明原因
  └──────────────┘
```

模組結構：
- **trust.py** — 信任度系統：基於歷史互動累積，建議結果好 → 升，害隊友受傷 → 降
- **personality.py** — 個性系統：謹慎/魯莽/保護型/學者型...影響決策偏好權重
- **judgment.py** — 情境評估：分析當前戰場狀態（HP、威脅、資源），產出情境分數
- **tactics.py** — 戰鬥決策：綜合信任度 + 個性 + 情境 → 選擇動作（確定性邏輯）
- **dialogue.py** — 對話觸發：決定什麼時候插話、回應玩家建議（走 LLM）

**設計原則**：
- 戰術評估是確定性的（走 Bone Engine 資料），不用 LLM 選動作
- 個性表現和對話才走 LLM（透過 Narrator）
- 信任度存在 Episodic Memory，跨 session 持續累積
- 隊友可能拒絕玩家建議，並透過對話解釋原因

### Extension（延伸精靈）

處理規則外的即興情境：
- 玩家做了規則書沒寫的事
- 即興裁定（合理性判斷 + 臨時規則）
- 自定義物品/法術的平衡檢查

## 三層記憶系統

### Working Memory（Redis）

- 當前戰鬥狀態（先攻順序、HP、位置、持續效果）
- 當前場景資訊（環境描述、NPC 在場）
- 玩家暫存動作（多步驟操作的中間狀態）
- TTL：session 結束後清除

### Episodic Memory（PostgreSQL + pgvector）

- 冒險歷史（每個 session 的事件摘要）
- NPC 互動記錄（對話、好感度、交易）
- 玩家選擇歷史（用於個性化敘事）
- 向量索引用於相似場景檢索

### Semantic Memory（Qdrant RAG）

- D&D 2024 (5.5e) 規則全文檢索
- 世界設定知識庫
- 怪物/法術/物品百科
- 用於 Narrator 和 Extension 的規則參考

### Context Assembler

記憶系統的統一介面，根據當前情境組裝 LLM prompt context：

```
Context = Working Memory (當前狀態)
        + Episodic Memory (相關歷史，向量檢索)
        + Semantic Memory (相關規則，RAG)
        + System Prompt (敘事風格設定)
```

## 怪物 AI 系統（Phase 6）

```
combat_ai/
├── behavior_tree.py   # 行為樹框架
├── goap.py            # Goal-Oriented Action Planning
├── tactics.py         # 通用戰術邏輯
└── monster_ai/
    ├── goblinoid.py   # 哥布林族：狡猾、群體戰術
    ├── undead.py       # 不死族：無畏、持續壓迫
    ├── beast.py        # 野獸：本能反應、逃跑閾值
    └── boss.py         # Boss：多階段、技能組合
```

採用 Behavior Tree + GOAP 混合架構，讓怪物有「智能」而非只是隨機攻擊。
