> **策略**：先以本地 CLI 打磨核心遊戲體驗，確認規則手感、遊戲節奏、資訊呈現皆到位後，再搬上 Telegram。

## 1. 技術選型：Textual TUI

### 為什麼不用純 print？

| 痛點 | 說明 |
|------|------|
| 資訊被洗掉 | 純 print 的 HP、地圖等關鍵資訊會隨滾動消失 |
| 無法分區 | 地圖、狀態、敘事、輸入全混在一起 |
| 輸入不友善 | 沒有自動補全、歷史指令、快速鍵 |

### 為什麼選 Textual？

- **Python 原生**：和 Bone Engine 同語言，無 FFI 開銷
- **分區佈局**：CSS-like 排版，可自由組合面板
- **Rich 生態系**：表格、語法高亮、進度條、Markdown 渲染
- **非同步原生**：基於 asyncio，和未來的 LLM 呼叫天然相容
- **可測試**：Textual 內建 pilot API，可以寫 UI 自動化測試
- **建議版本**：`textual >= 0.50`；相依新增 `textual`、`rich`

### 各遊戲階段的面板變化

| 遊戲階段 | 左上（地圖面板） | 右上（狀態面板） | 中間（敘事面板） | 下方（指令區） |

| **角色卡** | 角色全身資訊 | 裝備/法術列表 | — | 返回遊戲 |

## 3. 輸入設計：指令 + 選單混合制

### 設計原則

- **門檻低**：新手可以靠數字選單玩
- **效率高**：熟手可以直接打指令跳過選單
- **TTRPG 感**：保留打字互動的自由度
- **為 Mimic 鋪路**：自然語言輸入最終可走 LLM 解析

### 戰鬥回合互動流程

```
【回合 2 — Wizard 的回合】
可用動作：
  1. 攻擊（武器）
  2. 施放法術
  3. 移動
  4. 閃避
  5. 脫離
  6. 幫助
  7. 查看狀態
  8. 結束回合

> 2

選擇法術：
  1. 火球術 (3 環)  — 8d6 火焰，20呎球    [剩餘: 2 個 3 環]
  2. 魔法飛彈 (1 環) — 3 發自動命中       [剩餘: 4 個 1 環]
  3. 冰刃之壁 (戲法) — 1d10 冰冷
  4. ← 返回

> 1

選擇目標位置 (x,y)：
  提示：Goblin1 在 (3,4)，Goblin2 在 (5,4)，中心 (4,4) 可覆蓋兩者
> 4,4

✦ Wizard 施放【火球術】→ 中心 (4,4)，半徑 20 呎
  Goblin1：敏捷豁免 DC 15 → 擲出 8 → 失敗！→ 28 火焰傷害 → 擊殺！
  Goblin2：敏捷豁免 DC 15 → 擲出 17 → 成功！→ 14 火焰傷害
  Fighter：敏捷豁免 DC 15 → 擲出 12 → 失敗！→ 28 火焰傷害
  ⚠ 友軍傷害！Fighter 受到 28 點火焰傷害 (45→17 HP)
```

### 快捷指令（熟手模式）

跳過選單直接輸入：

```
> attack goblin1                     # 用裝備的武器攻擊
> cast fireball 4,4                  # 施放火球術到座標
> move 3,2                           # 移動到座標
> look                               # 查看環境
> status                             # 查看完整角色狀態
> status wizard                      # 查看特定角色
> spells                             # 查看法術列表
> map                                # 重新渲染地圖
> help                               # 指令說明
> save                               # 手動存檔
> quit                               # 存檔並離開
```

### Tab 補全

- 目標名稱：`gob<Tab>` → `goblin1`
- 法術名稱：`cast fire<Tab>` → `cast fireball`
- 指令名稱：`att<Tab>` → `attack`

## 4. 資訊層級管理

文字遊戲最大的 UX 問題是**資訊過載**。分三層處理：

### Always Visible（常駐顯示）

永遠顯示在狀態面板，不會被滾動洗掉：

- 隊伍 HP / AC / 狀態異常
- 當前遊戲階段（探索 / 戰鬥 / 休息）
- 戰鬥中：回合數、當前行動角色、先攻順序

### On Demand（按需查看）

玩家主動查詢才顯示：

| 指令 | 顯示內容 |
|------|---------|
| `look` | 環境詳細描述、可互動物件 |
| `status` | 完整角色面板（屬性/技能/裝備/法術欄位） |
| `spells` | 已準備法術列表 + 剩餘欄位 |
| `inventory` | 背包物品 |
| `conditions` | 所有生效中的狀態異常 + 剩餘回合 |
| `initiative` | 完整先攻順序 |

### Scrollable Log（可回捲紀錄）

敘事面板支援向上捲動，但不佔用其他面板空間：

- 戰鬥結果紀錄
- 骰子結果明細
- 環境互動描述
- NPC 對話（未來）

## 5. 視覺回饋設計

### 色彩方案

| 元素 | 顏色 | 用途 |
|------|------|------|
| 傷害數字 | 紅色 | 受到傷害 |
| 治療數字 | 綠色 | HP 恢復 |
| 骰子結果 | 黃色 | 吸引注意力 |
| 重擊 | 粗體紅色 | 自然 20 |
| 大失敗 | 灰色 | 自然 1 |
| 系統訊息 | 暗灰色 | 不干擾遊戲氛圍 |
| 狀態異常 | 紫色 | 中毒/魅惑等 |
| 法術效果 | 青色 | 法術相關敘述 |

### 地圖符號

```
@ = 玩家角色（依職業上色）
G = 哥布林（紅色）
S = 骷髏（灰色）
# = 牆壁
. = 地板
~ = 水
^ = 陷阱（已發現）
D = 門（關閉）
| = 門（開啟）
? = 未探索
★ = 目標/寶箱
```

### HP 條顯示

```
[██████████] 100%     綠色
[████████░░]  80%     綠色
[██████░░░░]  60%     黃色
[████░░░░░░]  40%     橙色
[██░░░░░░░░]  20%     紅色
[░░░░░░░░░░]   0%     暗紅色
```

## 6. 遊戲模式：單人控全隊

本地 CLI 版本先做**單人控制整隊**（類似 Baldur's Gate）：

- 玩家控制所有 PC 的行動決策
- 不需處理多人同步、等待其他玩家
- 可專注打磨遊戲節奏和規則手感
- 未來加入 Companion Gremlin 時，可選擇讓 AI 接管部分隊友
